## 前提となる実測事実(詳細: eval/REPORT.md「委譲検証」)

- 委譲のオーケストレーション費は呼び出し元 Claude Code 側の**固定 floor**(委譲先非依存 — Haiku 対照で確認)。第5ラウンドで分解すると **セッション起動費 S ≈ $0.10(委譲件数に非依存)+ 1件あたり作業費 T ≈ $0.03**。単発委譲は S+T ≈ $0.11〜0.18/件で損益分岐 baseline ≈ $0.15。
- **バッチ委譲で起動費が償却される(第5ラウンド課題1=合格)**: 1セッションに独立 N 件を委譲すると S が件数で割られ、1件あたり実効費は限界費 T($0.03)へ漸近。実測(batch-trio、独立3件)で 1件 $0.064 = 単発 floor($0.132/件)比 **−51.6%**・baseline 比 −45.3%。損益分岐は2段書き(**単発 $0.11〜0.18/件 / バッチ時 $0.06〜0.08/件**)。単発では負ける doc-sync 級もバッチ内なら黒字化。
- 節約の現実的上限は **−30〜50%帯**(第4ラウンドで「baseline $1超 → −80%級」を反証)。機械的で規則を記述できるタスクは Sonnet がスクリプトに畳むため、baseline はファイル数に線形でスケールしない(「スクリプト化の天井」)。
- submit/wait は **headless では「使わない機能」と断定(第5ラウンド課題3=不成立)**: 単一タスクで純損(+33%)に続き、本来の使い所(委譲向き A ‖ 委譲不適 B)でも B≪A のため net 壁時計は縮まず(async-pair 実測: 非同期 424s > 同期 384s)。overlap 機構は動く(B を A の裏に隠せる)が、理論上限 min(A,B)=B≈24s がローカル推論の分散(±31s)に飲まれる。コスト −12% はキャッシュ会計の副作用(n=1・再現未検証)で壁時計短縮の対価ではない。価値は対話セッションの並行作業のみ。
- `--check` つき委譲は品質退行ゼロ(6/6 PASS、第3ラウンド)。`--resume` は実装済みだが実効未測定(課題1)。
- stats を委譲判断に接続済み(`lh stats --by-kind --json` に pass rate、SKILL/AGENTS に事前チェック: graded≥3 かつ rate<50 なら委譲しない)。**効果の定量測定は fail 過半の kind が実データにたまってからの宿題**。インストール版 SKILL.md への事前チェック文の同期は doc-sync 対象として残る。
- 実時間は 3.7〜6.8 倍(うち委譲構造は ~1.5 倍、残りはローカル推論速度 — 課題2)。
- 運用注意: eval 実行はサブエージェントからは権限クラシファイアに拒否される — **ユーザー承認を得てメインセッションから直接実行**。再実行前に現行結果が上書きされないか確認し、必要なら `eval/results/*.roundN.*` へ退避。eval/run.ts を `bun -e` で import しない(`import.meta.main` が発火して全タスク eval が走る — 確認は grep/Read で)。

## 課題1: --resume の実効測定

ローカルが1回差し戻されやすい厳密フォーマット系タスク(no-repro 変種)を作り、「`--resume` 追撃 vs フル再委譲」のコスト・成功率を比較。合格条件: resume 側が安く、かつ最終 PASS。第3ラウンド以降 check 失敗の実例が出ていないため未測定のまま繰り上げ(実装・スモークは完了済み)。

## 課題2: ローカル推論の実時間短縮

委譲の最大ペナルティはコストでなく実時間(残り ~2.5〜4.5 倍分は推論速度)。`num_ctx` 65536 は委譲用途には過大の可能性——32768/16384 での tok/s を実測し、既存 eval 5タスク程度で PASS 退行がないか回帰。ほか量子化変更の検討。対話利用の採用障壁はここ。

## 課題3: 評価の未カバー軸

曖昧仕様で委譲先が「黙って推測」せず質問/報告できるか(no-repro に次ぐ危険故障モードの検査)を最優先に、非TS言語・マルチターン・意図的に長考を誘発するタスクでの thinking watchdog 実効性(第4ラウンドからの持ち越し)を順次。

## やらないこと(実測が「無駄」と示したもの)

- lh 側での floor 削減の追求(floor は呼び出し元の構造費 — Haiku 対照 H1 で確定)。
- `--worktree` パッチモード(見送り確定: 並行編集事故ゼロ+--check で動機解消)。
- 「−80%級」を狙う重量級 fixture の再設計(スクリプト化の天井により、機械的タスクで baseline $1 超は構造的に作れない)。
- headless / 自動委譲フローでの submit/wait 推奨(第5ラウンドで不成立確定: 単一でも A‖B でも net 壁時計短縮なし。対話セッションの並行作業限定)。
