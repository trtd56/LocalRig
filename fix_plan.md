## 前提となる実測事実(詳細: eval/REPORT.md「委譲検証」)

- **ドル絶対額は Claude Code バージョンの会計に依存する(第6ラウンドで確定)**: 2.1.77→2.1.202 でシステムプレフィックス 10,927→35,592 tok・キャッシュ書き実効単価 $3.75→$6/M。第1〜5ラウンドの $ 値は 2.1.77 会計として読む。**コスト実測は「同日・同バージョン・キャッシュ warm 統制」の同時比較でのみ成立**(共有プレフィックス 24,010 tok の cold/warm でターン1が ~$0.14 ずれる。ネスト claude の version は transcript jsonl の `version` フィールドで確認できる)。
- 委譲のオーケストレーション費は呼び出し元 Claude Code 側の**固定 floor**(委譲先非依存 — Haiku 対照で確認)。第5ラウンド分解で **セッション起動費 S(件数非依存)+ 1件あたり作業費 T**(2.1.77 会計で S≈$0.10・T≈$0.03、単発 S+T≈$0.11〜0.18/件・損益分岐 baseline ≈$0.15)。
- **バッチ委譲で起動費が償却される(第5R)+ 束ね方は一級 `lh batch` 一択(第6R)**: 2.1.77 会計で 1件 $0.064(単発比 −51.6%)。2.1.202 会計の同日 3-way(batch-trio)では **手組み 3×`lh -p` ≈$0.60+ = baseline($0.446)比 +27% の赤字 / 一級 `lh batch` ≈$0.39〜0.42 = −6〜13% の黒字**。プレフィックス肥大で「N 回呼んで N 回処理する」ターン数自体が罰金になったため。加えてバッチは実行・check・修復・永続化がローカル側で完結し、**呼び出し元が死んでも委譲済みタスクは完遂・逐次保存される**(実測: 手組みは環境死で 1/3 委譲のまま FAIL、バッチは 2 run とも 3/3)。
- 節約の現実的上限は **−30〜50%帯**(第4ラウンドで「baseline $1超 → −80%級」を反証)。機械的で規則を記述できるタスクは Sonnet がスクリプトに畳むため、baseline はファイル数に線形でスケールしない(「スクリプト化の天井」)。
- submit/wait は **headless では「使わない機能」と断定(第5ラウンド)**: 単一タスクで純損(+33%)、A‖B でも B≪A のため net 壁時計は縮まない。価値は対話セッションの並行作業のみ。
- `--check` つき委譲は品質退行ゼロ(6/6 PASS、第3ラウンド)。`--resume` は実装済みだが実効未測定(課題1)。
- stats を委譲判断に接続済み(`lh stats --by-kind --json` に pass rate、SKILL/AGENTS に事前チェック: graded≥3 かつ rate<50 なら委譲しない)。**効果の定量測定は fail 過半の kind が実データにたまってからの宿題**。インストール版 `~/.claude/skills/delegate-local/SKILL.md` は 2026-07-07 に原本と同期済み(以後も原本更新時に cp で同期する——メインセッションからの cp は通る)。
- 実時間は 3.7〜6.8 倍(うち委譲構造は ~1.5 倍、残りはローカル推論速度 — 課題2)。ローカル壁時計の run 間分散は従来想定(±10%)より大きい実測点あり(同一バッチ 792s vs 1689s = 2.1 倍)。
- 運用注意: eval 実行はサブエージェントからは権限クラシファイアに拒否される — **ユーザー承認を得てメインセッションから直接実行**。再実行前に現行結果を `eval/results/*.roundN(-attemptM|-envfail).*` へ退避。eval/run.ts を `bun -e` で import しない(`import.meta.main` が発火して全タスク eval が走る — 確認は grep/Read で)。
- **間欠 ConnectionRefused(未解決の環境問題)**: ネスト claude が「`lh` を呼んだ Bash の完了直後の初回 API コール」で間欠的に `ConnectionRefused` 死する(内部リトライ ~3分で断念、`is_error:true`)。外部到達性は常に正常=プロセス局所。sandbox・cmux シム・アイドル長は無関係、根本原因未特定(EDR=Falcon 容疑)。**対処**: 死んでもローカル成果と check は残る(逐次保存)ので、`lh sessions`/`lh poll --json` で確認し feedback を事後記録、eval はアーカイブ→再実行。PASS/FAIL 判定は claude の `is_error` でなくローカル成果物+verify で行う。

## 課題1: --resume の実効測定

ローカルが1回差し戻されやすい厳密フォーマット系タスク(no-repro 変種)を作り、「`--resume` 追撃 vs フル再委譲」のコスト・成功率を比較。合格条件: resume 側が安く、かつ最終 PASS。第3ラウンド以降 check 失敗の実例が出ていないため未測定のまま繰り上げ(実装・スモークは完了済み)。

## 課題2: ローカル推論の実時間短縮 + ローカル壁時計の分散の解明

委譲の最大ペナルティはコストでなく実時間。(a) `num_ctx`(現既定 32768)の 16384 での tok/s と PASS 退行を回帰、量子化変更の検討。(b) **第6ラウンドの新規データ**: 同一3タスクのバッチをアーカイブ6 run で比べると、ローカル合計は 333s〜1688s(5倍)。per-task の `durationMs`(session JSON の tasks[] に記録済み)で分解すると、docs 級 80→993s・types 級 179→865s と**単一タスクの分散が支配的**で、「一級 batch が手組みより構造的に遅い」は不成立(最速 run 333s は手組み 3連発 355s と同等)。ターン数はほぼ同じで per-turn 時間だけ伸びる run があるため、Ollama 側の状態(KVキャッシュ・メモリ圧・サーマル)を疑って tok/s を run 毎に記録するのが先決。

## 課題3: 評価の未カバー軸

曖昧仕様で委譲先が「黙って推測」せず質問/報告できるか(no-repro に次ぐ危険故障モードの検査)を最優先に、非TS言語・マルチターン・意図的に長考を誘発するタスクでの thinking watchdog 実効性(第4ラウンドからの持ち越し)を順次。

## 課題4: blog 公開前の n=3(測定プロトコル改訂版)

見出し数値は全セル n=1。公開前に該当セル——特に **batch-trio(baseline / 手組み / 一級 batch)と async-pair**——を3回ずつ再測して幅を併記。**必ず同日・同 CLI バージョン・warm 統制で全アームを連続実行し、バージョンを記録すること**(第6ラウンドの教訓: バージョン跨ぎの $ 比較は無効。ローカル壁時計は ±2 倍の分散があるため中央値で報告)。

## やらないこと(実測が「無駄」と示したもの)

- lh 側での floor 削減の追求(floor は呼び出し元の構造費 — Haiku 対照 H1 で確定)。
- `--worktree` パッチモード(見送り確定: 並行編集事故ゼロ+--check で動機解消)。
- 「−80%級」を狙う重量級 fixture の再設計(スクリプト化の天井により、機械的タスクで baseline $1 超は構造的に作れない)。
- headless / 自動委譲フローでの submit/wait 推奨(第5ラウンドで不成立確定)。
- 手組み逐次委譲(1件ずつ `lh -p`)を複数タスクに使うこと(第6ラウンドで赤字確定 — `lh batch` に一本化)。
