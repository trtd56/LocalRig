# タスクルーティング・プレイブック(実履歴分析版)

過去の Claude Code / Codex / 通常チャットの利用履歴からタスクを抽出・分類し、
「どの種類のタスクを、素のOllama / pi-worker / LocalRig(lh) / フロンティア(Claude・Codex)のどれに投げるべきか」を整理したもの。

- 分析日: 2026-07-11〜12
- データ1(定量): Claude Code `~/.claude/history.jsonl` 2,082件(2025-10〜2026-07) + Codex `~/.codex/sessions/` rollout 3,584件(2025-08〜2026-07、dedup後) = **計5,666プロンプト**。7並列サブエージェントで「タスク種別 × ローカル委譲適性(L1/L2/L3/F)」に全件分類
- データ2(定性): 通常チャット(claude.ai)の直近数ヶ月の会話を「モデル能力がボトルネックか、それ以外(検索・ツール・コンテキスト)がボトルネックか」で仕分け

## 統合判定ルール(これだけ覚えればよい)

> **「入力に必要な情報が全部揃っていて、出力が1〜2ターンで完結するテキスト操作」はローカルに逃がせる。**

1. ツール不要・情報が揃った1〜2ターンのテキスト操作(メール・翻訳・整形・推敲・命名・ブレスト叩き台・定番CLI知識Q&A) → **素のOllama**
2. コードベースの位置特定質問・長大ログの圧縮 → **pi-worker**(scout / distill)
3. ファイル編集を伴う機械的タスクで成功が `--check` で機械検証できる → **lh**(2件以上は `lh batch`)。ただし sed/スクリプトで済むならスクリプト化が最速
4. Web検索前提の調査・反復デバッグ・巨大コンテキスト・設計判断・思考パートナー → **フロンティア**

ボトルネックの見極めが本質: ローカルで体感差が出る原因は「モデル能力」だけでなく「検索ツール統合・メモリ・コンテキスト長」であることが多い。能力ボトルネックでないタスクは27B級で差がほぼ出ない。

## ルート別ガイド

### 1. 素のOllama(直接 /api/chat・ollama run) — L1帯

ファイル操作・検証が不要な単発テキスト処理。オーケストレーション費ゼロ。コーディング履歴(約70件+text_processing L2)と通常チャット履歴の両方で頻出し、**通常チャット側では日常タスクの3〜4割がここに該当**する見立て。

コーディング履歴由来:
- 翻訳・要約(「英語は翻訳して」「トランスクリプトの要約」)
- 命名・文案(「アプリの名前を考えて」「YouTubeタイトルと概要文」「X投稿文」「Slack文面」)
- コミットメッセージ・PRメッセージの下書き(diffを渡す)
- CSV化・フォーマット整形
- 「ジュニアレベル解説」の下書き(最終チェックは人間)

通常チャット履歴由来(定性分析):
- **日本語ビジネス文面の作成・調整(頻度: 高)**: イベント参加者向けメール、挨拶文の言い換えNパターン、トーン調整。定型度が高く27B級で十分。「■」「【】」等の日本語ビジネス慣習も指示すれば再現可
- **テキストの変換・整形(頻度: 中)**: Markdownテーブル→箇条書き、X告知文の短縮。完全に局所的で差はほぼゼロ
- **定番CLI/Gitトラブルの知識Q&A(頻度: 中)**: unstaged changesでpullがブロック、SSH direct vs proxy比較など、よく知られた領域の知識質問
- **翻訳(頻度: 中)**: 告知PDFの和訳など。27Bなら実用品質
- **原稿の局所推敲**: 与えたテキストの言い回し改善はOK。ただし記事全体の再構成(トーン統一・構成変更)のような長文一貫性が要る作業はフロンティア側
- **軽めのブレスト叩き台**: 適用事例案・題材案の発散フェーズ。網羅性・意外性は落ちるが叩き台として機能

公開物(ブログ・SNS・メール送信)は下書き止まりにして人間が仕上げる。

### 2. pi-worker(Pi経由) — 読み取り系・前処理系

pi 0.79.3 のループ + check/scope/scout/distill。**編集系よりも scout / distill が履歴と噛み合う**。

- `pi-worker run --scout`: コードベース位置特定質問(qa_explain L2 ≈105件が該当)。
  「〇〇はどこで定義されてる?」「起動コマンドは?」「このシステムプロンプトはどこ?」→ 引用付き回答
- `pi-worker distill`: **debug 606件の前処理**。Traceback・ビルドログ・長大エラーの圧縮と要点抽出をローカルで行い、フロンティアに渡すトークンを削減(貼り付けデバッグは履歴最大のクラスタ)
- scope guard付き編集(`--allow-path`/`--protect-path`): 業務リポジトリで保護境界が欲しい小修正

注意: 実測はまだ薄い(smoke n=1、scout citation precision 90%)。本格運用前に代表タスクで追加評価を。

### 3. LocalRig lh — ファイル編集を伴う機械的タスク(委譲の本命)

実証済み: 改良版委譲で baseline比 −22%(品質退行ゼロ)、mass-migration級 −44%、束ねる場合は一級 `lh batch` 一択(手組み比 ≈−30%)。floor ≈$0.10-0.15/呼び出しなので **baseline $0.15未満の単発は委譲しない**。

履歴から見えた定常的な委譲候補(L2帯 ≈730件):

- **mechanical_edit(151件)**: 文言・設定値・カラム順・表記ゆれ修正、計画ファイル整頓、gitignore追加
- **docs(75件)**: README更新・同期、実験記録md、日報、仕様ドキュメント ※現状 `lh stats` で docs kind が sandbox タイムアウトで 0/2 fail — 時間予算とsandbox設定を直してから
- **ui_game(60件)**: 「速度を落として」「横幅を広げて」「カメラ左右反転」級の数値・スタイル微調整(子供ゲーム・個人ツールで高頻度)
- **small_code(50件)**: ffmpeg一括変換、プログレスバー追加、ストリーミング出力化、単機能スクリプト
- **ops_env の一部(144件中)**: 「実験記録をまとめてcommit」「gitignore追加」「mp4→mp3一括変換」

運用ルール(委譲検証第1〜6ラウンドの結論):

1. 必ず `--check` を付ける(受け入れゲートをローカル機械検証に置く。検証者の検証は受け入れゲートより浅い)
2. 2件以上あれば `lh batch`(1セッション束ねで1件あたり費用が約半減)
3. チケットは10行以内・自己完結(委譲前のread ≤2ファイル、ファイル内容の貼付・解法の記述は禁止)
4. `lh stats --by-kind` で graded≥3 かつ pass<50% の kind は委譲しない(現在: research が gate=block)
5. **委譲 vs スクリプト化の三択**: 27Bに委譲できるほど機械的なタスクはスクリプト化できることが多い(mass-migrationの教訓)。sed/一括処理で済むならそれが最速・最安

### 4. フロンティア維持(Claude / Codex / 通常チャット)

コーディング履歴由来:
- **debug の大半**: 環境・認証(GCP/Zoom/Zitadel)・非同期・音声系の深いデバッグ(F 262 + L3の多く)
- **data_ml(399 F)**: Kaggle自律実験ループ、精度改善戦略、DSPy最適化
- **design(220 F)**: PLAN.md/ACCURACY_PLAN駆動の計画立案・書き直し
- **review の差分レビュー(201 F)**: 「developmentとの差分をレビュー」定型。一次観点出し(nit・型・命名)だけローカルに前置きする余地はあるが、最終判断はフロンティア
- **research**: lh実測でも gate=block。網羅性・引用忠実性がローカルの弱点

通常チャット履歴由来(定性分析) — 差が大きく出る領域:
- **Web検索前提の情報収集**: 最新モデル情報、プロバイダ比較、SDK調査、企業マップ、競合ポジショニング検証。モデル能力というより検索ツール統合の問題だが、検索結果の取捨選択・矛盾検出の質も効くため、ローカルに寄せると体感差が大きい
- **反復デバッグ込みのコーディング**: マルチターンのコード修正は差が顕著(OpenCodeベンチのQwen3.6-27B体感とも一致)
- **巨大コンテキスト処理**: 数百ページ級の資料分析。コンテキスト長と長文注意力の両方でローカル不利
- **深い思考パートナー系**: キャリア熟考、過去会話横断の自己分析。推論の質+メモリ/検索機能の両面でクラウド依存
- **統合的サーベイ**: 一次ソース検証込みの調査(ガードレール分類等)
- **長文一貫性が要る執筆**: 記事全体の再構成・トーン統一・構成変更

## 2つの分析の整合

- コーディング履歴(定量)では即委譲可能(L1+L2)は分類可能タスクの約2割、通常チャット(定性)では日常タスクの3〜4割がローカル可 — **母集団が違うだけで矛盾しない**。コーディング履歴は反復デバッグ・実験・設計などF級が構造的に多く、チャット履歴は文面作成・翻訳・整形などL1級が多い
- 両分析の結論は同じ判定則に収束する: 「入力が揃っていて1〜2ターンで完結するテキスト操作」= ローカル、「検索・エージェント・長文・反復修正」= クラウド
- 運用イメージ: メール・整形・翻訳・推敲あたりを日常的にローカル(素のOllama)へルーティングし、コーディングの機械的L2断片は lh batch へ、検索・エージェント・長文だけクラウドに残す

## ワークフロー上の発見

- **全プロンプトの26.5%が「続けて」系の追撃** — 伴走型オーケストレーションが基本スタイル。ローカル委譲はこのスタイルと直交するので、「委譲できる形にチケット化してから lh batch に束ねる」変換が委譲活用の前提
- Claude Code は Kaggle 自律実験の監督、Codex は業務リポジトリ(ais-hayabusa)のレビュー・PRコメント対応が最大クラスタ、と使い分けが既に成立
- 反復定型パイプライン(stack_reachy 動画制作、daily_tech_research、norma_jeane_v2 計画レビュー往復)は個々のステップがL2級 — バッチ委譲かスクリプト化の好対象
- 履歴には自前評価ハーネスが注入したプロンプト(Titanic分析等)が混入しており、data_ml F はやや過大推定

## 注意事項

- 分類は各プロンプト単体の判定(n=1セッション文脈は未考慮)。L3の多くは「文脈を渡せばL2化する」ものを含む
- コスト数値は 2026-07 時点・Claude Code 2.1.202 会計。CLIバージョン間のドル比較は無効(第6ラウンドの教訓)
- 日常テキスト系の受け皿はメインの Qwen3.6 27B に限らず、EVO-X2 上の 35B-A3B 等の軽量高速モデルでも成立する(テキスト一発処理はエージェント調整値に依存しないため)
- 次の一手の候補: この判定則を「ルーティング判定プロンプト」(タスクを受けて local/cloud を振り分ける小分類器)としてローカルモデル自身に載せる
