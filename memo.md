# 開発メモ: 試行錯誤の記録

LocalRig 実装時(2026-07-03)に踏んだ罠・覆った判断・設計の分岐点の記録。
確定した設計と出典は [DESIGN.md](./DESIGN.md)、評価結果は [eval/REPORT.md](./eval/REPORT.md) を参照。

## 1. 環境調査でつまずいた点

- **`ollama show qwen3.6-27b-mtp:local` が失敗する**。`ollama list` には出るのに show では `qwen3.6-27b-mtp:latest not found` になる(タグ解決の不具合、ドット入りの名前が原因の可能性)。同一IDのエイリアス `qwen36-27b-mtp:latest` なら通る。ハーネスの既定モデル名はこちらを採用。
- モデルは 256k コンテキスト宣言だが、Ollama は `num_ctx` 未指定だと小さい既定値で**静かに切り詰める**(OpenCode のドキュメントにも「4k のままだとツールスキーマが切れて tool calling が静かに壊れる」とある)。全リクエストで明示指定する設計にした(既定 65536、64GB RAM 機で KV キャッシュ込みでも余裕)。
- 事前の curl テストでネイティブツールコールが「並列コール+thinking 分離」まで一発で動くと確認できたため、当初想定していた「テキストパース主体」の設計を「ネイティブ第一+テキストフォールバックは保険」に反転。実際、評価4タスクでフォールバックの発動は0回だった。

## 2. 調査で覆った初期判断

3実装(pi / OpenCode / qwen-code)の調査は初期設計を4点で覆した:

| 初期判断 | 覆った後 | 根拠 |
|---|---|---|
| temperature 0.2 / top_p 0.9(Modelfile の既定値をそのまま採用) | **0.6 / 0.95 / top_k 20** | Qwen3.6 モデルカード: thinking モードで低温度は**反復ループを誘発**。コーディング推奨値は 0.6/0.95/20 |
| num_predict 8192 | 16384 | thinking が長くなると 8k では本文が切れる |
| edit は exact match + replace_all のみ | **5段マッチングカスケード**(exact→Unicode正規化→行トリム→エスケープ正規化→ブロックアンカー) | OpenCode「弱いモデルに最もレバレッジが効く単一の設計」。pi も NFKC/スマートクォート正規化で同じ結論 |
| 切断応答(done_reason=length)は単に継続を促す | 切断応答中の **edit/write は実行拒否**して再発行を要求 | qwen-code: 途中で切れた JSON でファイルを破壊する事故の予防 |

他に調査で追加したもの: bash 全出力の一時ファイルスプール(切り詰めても grep で辿れる)、コンパクション失敗ガード(空サマリ→中止、圧縮後にトークンが**増えた**ら破棄)、最大ステップ到達時の「ツール禁止・テキストのみ」強制サマリ注入(OpenCode の MAX_STEPS_PROMPT)。

## 3. ローカル特有の設計分岐: プレフィックス KV キャッシュ

途中で気づいた最重要制約。Ollama はプロンプトの前方一致で KV キャッシュを再利用するため、**古いメッセージを1つでも書き換えると全プロンプト再評価**になる。27B で 50k トークンの再評価は数分かかり、毎ターンやると実用にならない。

これで決まった設計:
- システムプロンプトはセッション中**完全固定**(動的情報を入れない)
- todo リマインダーは**末尾に追記**(変更時のみ)。システムプロンプト書き換え方式は却下
- プルーニングは毎ターン少しずつではなく、**75% 到達時に一括**(どうせ1回で壊れるならまとめて回収し、発生頻度を下げる)
- thinking の履歴再送はしない。qwen-code は DashScope 向けに preserve_thinking するが、ローカルでは「コンテキスト消費」と「古いメッセージの書き換え禁止」の両制約からストリップ一択(Ollama のテンプレートも最終ターン以外は描画しない)

## 4. 統合時に踏んだ罠

- **サブエージェントが devDependencies を「掃除」した**: ツール実装エージェントが `bunx tsc` の副作用で node_modules ができたと勘違いし、私が意図して入れた typescript/bun-types を package.json ごと巻き戻した。並行作業では「どこまでが自分の副作用か」の境界を指示に含めるべきだった。→ 再インストールで復旧
- **評価フィクスチャが型チェックとテストを汚染する**: fixture には「わざと失敗するテスト」「存在しないモジュールの import」が含まれるため、(a) `tsc` がフィクスチャでエラー → tsconfig の include を `src/`, `test/`, `eval/run.ts` に限定、(b) `bun test test/` が**部分文字列マッチ**で `eval/tasks/*/fixture/test/` まで拾う → `bunfig.toml` の `[test] root = "test"` で解決
- **フォールバックパース後の二重描画**: `<tool_call>` テキストから回収したコールを tool_calls に足すだけだと、本文にタグが残ったまま履歴再送時に二重描画される → パース成功時は本文からブロックを除去
- **最終イテレーションでのツール実行**: 「最大ステップ到達、テキストのみで」と注入しても、モデルがツールコールを返したら実行してしまうバグ → wrapUp フラグで実行前に return
- **eval ランナーの summary 上書き**: `--task` 指定の単発実行が全体の summary-*.json を上書きする。ログ(タスク別 .log)は無事なので REPORT.md に手動集約した。ランナーを直すならタスク別ファイル化かマージ書き込み

## 5. 評価で分かったこと

- 4タスク(バグ修正/機能追加/リファクタ/複合難タスク)で**ハーネス+Qwen3.6 も Claude Code (Sonnet) も 4/4 PASS**。差は所要時間のみ(2〜4.7倍、最長178秒)
- 修復パイプライン・ループ検出・フォールバックは**一度も発動しなかった**。Qwen3.6 のネイティブツールコールは想定よりずっと堅牢(temp 0.6 設定の寄与もありそう)。これらの機構は「より弱いモデル・低量子化・長時間セッション」向けの保険として残す
- hard-multi での軌跡(todo計画→探索→編集→検証→正確な報告)は Claude Code と同質。**システムプロンプトは約600トークンで足りた**。pi の「frontier モデルに1万トークンのプロンプトは不要」という主張は、少なくとも Qwen3.6 27B クラスにも当てはまる
- 未検証の残課題: 大規模リポジトリでの長時間セッション(prune/compact の実地発動)、修復経路の実戦頻度、REPL モードの体感品質

## 6. サブエージェント分業の所感

インターフェース契約(types.ts + agent.ts の import 署名)を**先に固めてから**3体を並行起動したのが効いた。競合ゼロ、手戻りは devDependencies の件のみ。調査で判明した改善(edit カスケード等)は走行中のエージェントに SendMessage で追送して取り込めた。「契約先行 → 並行実装 → 追送で仕様更新」は再利用できるパターン。
