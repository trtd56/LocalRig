# Podcast 台本: Claude Code の仕事を Local LLM に委譲すると安くなるのか

想定尺: 30〜40分  
形式: 1人語り

## 0. オープニング

こんにちは。AIチームの戸田です。

今回は、Claude Code から Local LLM にコーディングタスクを委譲すると、本当にトークンを節約できるのか、という話をします。

最近、手元の Mac で動く Local LLM を、Claude Code のサブエージェントのように使う実験をしていました。

発想としてはかなりシンプルです。

Claude Code に全部やらせると、当然 Claude 側の API コストがかかります。  
でも、機械的で検証しやすい作業だけ Local LLM に投げれば、Claude Code 側のトークンを減らせるのではないか。

たとえば、大量のリネーム、ボイラープレートの生成、テスト追加、README の同期のような作業です。  
こういうものは、フロンティアモデルに全部やらせるのは少しもったいない気もします。

なので、Local LLM に仕事を逃がす CLI を作って、実際にどれくらい安くなるのかを測りました。

結論から言うと、条件付きで節約できます。  
ただし、「Local に投げれば安い」という単純な話ではありませんでした。

実際には、単発委譲には固定費があり、軽いタスクでは逆に高くなります。  
品質面では、Local LLM の能力よりも、検証をどこで行うかが重要でした。  
そして、複数タスクをまとめる batch は効きましたが、非同期の submit/wait は headless ではほとんど効きませんでした。

今日は、このあたりの試行錯誤を、実験の流れに沿って話していきます。

---

## 1. 何を作ったか

まず、今回作っていたものから説明します。

手元で LocalRig というコーディングエージェントハーネスを作っています。  
Ollama 上の Qwen3.6 27B を、Claude Code のようなコーディングエージェントとして動かすための TypeScript と Bun の実装です。

単にモデルを呼ぶだけでは、Claude Code のようには動きません。

Local LLM は、ツールコールが崩れたり、edit の対象を微妙に間違えたり、長い thinking で空転したり、コンテキストが伸びたときに弱くなったりします。

そこで、いくつか補強を入れています。

たとえば、ツールコールの修復。  
モデルが少し壊れた JSON を出しても、可能なら直してツール実行に回します。

edit も、完全一致だけではなく、空白や引用符の揺れを吸収する多段マッチングにしています。

また、thinking が長くなりすぎたら中断して、いったん検証に進ませる watchdog も入れました。

さらに、既存ファイルを丸ごと write しようとしたときのガードレールも入れています。  
これは、Local LLM がファイル全体を書き直したときに、既存の export を落としてしまう事故があったためです。

こういう補強を入れたうえで、`lh` という CLI にしました。

Claude Code からは、たとえば次のように呼べます。

```bash
lh -p - --json --cwd /path/to/repo --kind rename --check "bun test"
```

標準入力で作業指示を渡して、Local LLM が実装し、最後に `--check` のコマンドを実行する、という形です。

この CLI を、Claude Code からサブエージェントのように呼べるようにした、というのが今回の実験の土台です。

## 2. まず LocalRig 単体の性能を測った

いきなり委譲の話に行く前に、まず LocalRig 単体でどれくらい解けるのかを測りました。

Claude Code が日常的にやるような小さな fixture を作り、同じタスク、同じ検証コマンドで、LocalRig と Claude Code を比較しました。

最初は4タスクでした。  
バグ修正、機能追加、リファクタ、少し複雑なマルチファイル修正です。

そこから、コードベース探索、実行時エラー診断、API移行、git workflow、プロジェクト規約遵守などを足していきました。

さらに、高難度タスクとして、大きめのコードベース探索、仕様書駆動の実装、並行処理バグ、挙動変更を伴う移行なども入れました。

最終的には20タスクまで増やして、LocalRig は 20/20 PASS まで来ました。  
Claude Code も同じく 20/20 PASS です。

ここで分かったのは、ハーネスをちゃんと作ると、Local LLM でも成果物品質はかなり出るということです。

一方で、速度はかなり違います。

Local LLM はだいたい数倍遅いです。  
複雑なタスクでは、Claude Code の5倍、場合によってはそれ以上かかることもありました。

なので、この時点で見えていた構図はこうです。

品質は出る。  
でも速くはない。  
ただし、Local 推論なので、Claude 側の API コストはかからない。

この状態で、ようやく本題です。

Claude Code が自分で実装する代わりに、この LocalRig に作業を委譲すると、Claude Code 側のコストは本当に下がるのか。

---

## 3. Claude Code から委譲してみる

まず、委譲ユースケースに近い6タスクで試しました。

README 同期、再現不能なバグ報告の triage、性能改善、型エラー修復、テスト作成、12ファイル横断のリネームです。

Claude Code には、「実装は `lh` に委譲してください」と system prompt で指示します。  
タスク本文は baseline と同じにして、条件だけを変えました。

結果は、かなり面白かったです。

12ファイル横断の rename-sweep は、大きく勝ちました。  
Claude Code 単独では $0.337 だったのが、委譲では $0.126 です。  
だいたい63%減です。

これは期待通りでした。  
多ファイルにまたがる機械的な作業なので、Claude Code 側の実装トークンを Local に逃がせています。

一方で、README 同期、doc-sync は負けました。  
Claude Code 単独では $0.081 だったのが、委譲すると $0.120 です。  
49%高くなっています。

これは最初は少し意外でした。

ただ、考えると自然です。  
委譲すると、Claude Code は作業指示書を書きます。  
Local LLM の結果を待ちます。  
返ってきた JSON を読みます。  
差分を確認します。  
最後に feedback を記録します。

この一連のオーケストレーション自体にトークンがかかります。

つまり、委譲は実装コストをゼロにするものではありません。  
タスク規模に比例する実装コストを、ほぼ固定のオーケストレーション費に変えるものです。

実測では、委譲したときの Claude Code 側コストは、だいたい $0.11 から $0.15 くらいに寄りました。

なので、baseline がそれより高いタスクでは得をします。  
逆に、数ターンで終わる軽いタスクでは、固定費に負けます。

ここで最初の認識が変わりました。

Local LLM 委譲は、「全部投げれば安い」ではありません。  
「固定費を超える作業だけ投げる」ものです。

もうひとつ面白かったのは、Claude Code が指示を半分くらい無視したことです。

「必ず委譲してください」と書いても、6タスク中3タスクでは `lh` を呼ばずに自分で実装していました。

ただ、その無視されたタスクは、損益分岐に近い軽めのものが多かったです。  
なので、ある意味では経済的に合理的な不服従でした。

このあたりから、委譲は自然言語で「常にやれ」と言えば済むものではなく、判断基準やゲートを設計する必要があると分かってきました。

## 4. 品質リスクの正体

次に、委譲をもっと強制するようにしました。

「最初の `lh` 呼び出しが返る前に、ファイルを編集してはいけない」  
このレベルまで system prompt を強めました。

すると、委譲の遵守率は上がりました。  
コストも baseline 比で23%くらい下がりました。

ただし、品質が落ちました。  
6タスク中1つが FAIL しました。

失敗したのは no-repro というタスクです。

これは、精巧な偽バグ報告が ISSUE.md に書かれていて、それを実際に再現しようとするタスクです。  
再現できなければ、コードは変更せず、TRIAGE.md の1行目に `verdict: not-reproducible` と完全一致で書く必要があります。

Local Qwen の判断自体は正しかったです。  
ちゃんと再現不能だと判断していました。

でも、TRIAGE.md の先頭に Markdown 見出しを書いてしまいました。  
つまり、1行目の完全一致条件を破りました。

問題はそのあとです。

Claude Code は、TRIAGE.md の意味内容を見て、「再現不能って書いてあるからOK」と判断し、`feedback pass` を記録してしまいました。

ここで分かったのは、委譲の品質リスクは Local LLM が間違えることだけではない、ということです。

むしろ危ないのは、呼び出し元の検証が、受け入れゲートより浅くなることでした。

Claude Code が自分で実装したときは、このタスクは PASS していました。  
でも委譲すると、作業者と検証者が分かれます。  
そして検証者である Claude Code が、厳密な verify ではなく意味で見てしまうと、偽 pass が起きます。

これはかなり重要な発見でした。

Local LLM の能力だけを見ていても、この故障モードは見えません。  
委譲というワークフローにしたことで、新しく生まれたリスクです。

## 5. `--check` を入れる

この対策として、`lh` に `--check` を入れました。

`--check` は、Local LLM の作業が終わった後に、`lh` 自身が受け入れコマンドを実行する仕組みです。

たとえば、`bun test` や `bash test/verify.sh` を渡しておきます。  
Local LLM の作業が終わったら、その場で check を走らせます。

失敗した場合は、Local LLM 側で修復リトライします。  
そして、結果を JSON に入れて Claude Code に返します。

Claude Code は、意味内容を頑張って読むのではなく、まず `check.exit_code` を見ます。  
そのうえで diff を確認し、feedback を記録します。

この形で再実測すると、6/6 PASS に戻りました。  
コストは baseline 比で22%減です。

面白いのは、修復ループは一度も発火しなかったことです。

つまり、効いたのは自動修復ではありません。  
受け入れゲートを、作業指示の中の曖昧な要件から、Local 側で実行される機械検証に移したことが効いていました。

`--check` はコストを下げる機能ではありません。  
品質を落とさないための機能です。

このあたりで、委譲の実用条件がかなりはっきりしてきました。

委譲するなら、厳密に検証できるタスクに限る。  
そして、受け入れコマンドを `--check` に渡す。  
Claude Code はその結果を見て、diff を確認してから feedback を記録する。

この流れが必要です。

## 6. Haiku に委譲するとどうなるか

ここまで来ると、もうひとつ気になることがあります。

この問題は、自作ハーネスや Qwen の問題なのか。  
それとも、委譲という構造そのものの問題なのか。

そこで、委譲構造は同じまま、委譲先だけ Haiku に差し替えた対照実験もしました。

結果として、Haiku は6/6 PASSでした。  
品質は良いです。

ただし、コストは baseline より16%高くなりました。

これはかなり分かりやすい結果です。

委譲には、Claude Code 側の固定オーケストレーション費があります。  
そこに、Haiku ワーカーの課金が乗ります。  
つまり二重払いになります。

なので、「委譲でコスト削減」が成立するのは、ワーカーの限界費用がほぼゼロのときだけです。

Local LLM は、この構図で黒字を出せる数少ない構成でした。

これは、モデル品質の話だけではなく、会計構造の話です。

## 7. 重いタスクなら大勝ちする、と思った

次に、もっと重いタスクなら大きく勝てるのではないかと考えました。

rename-sweep は12ファイルでかなり勝ちました。  
なら、40ファイルくらいの移行なら、もっと大きく勝てるのではないか。

そこで、mass-migration という fixture を作りました。

40ファイル、46箇所にまたがる API 移行です。  
`log.write(level, msg)` を `logger.emit({ level, msg, source })` に書き換えるタスクです。

ポイントは、`source` の正解値が各ファイル先頭のコメントにだけ書かれていることです。  
パスからは導出できません。

単純な sed だけでは解けないようにしたつもりでした。

rename-sweep から外挿すると、Claude Code 単独なら $1.1 から $1.5 くらいかかるのではないかと予想していました。

実測は $0.23 でした。

かなり外れました。

トランスクリプトを見ると、理由はすぐ分かりました。

Sonnet は46箇所を手で編集していません。  
コメントから正解値を機械抽出する Python スクリプトを書き、一括で書き換えていました。

つまり、baseline はファイル数に線形で増えません。

規則が書ける機械的タスクであれば、Sonnet は自分でスクリプト化して安く済ませます。

委譲アームは $0.129 で PASS しました。  
なので、44%減で勝ってはいます。

ただ、当初期待していた80%減のような世界ではありませんでした。

ここで見えた構造はかなり重要です。

Local LLM に安心して委譲できるほど機械的なタスクは、Sonnet がスクリプトで畳めるほど機械的なタスクでもあります。

なので、委譲には下限と上限があります。

下には固定費の壁があります。  
軽いタスクは固定費に負けます。

上にはスクリプト化の天井があります。  
重い機械的タスクは、そもそも Sonnet がスクリプト化して安く済ませることがあります。

このあたりから、判断は「委譲するか、自力でやるか」ではなくなりました。

実運用では、委譲する、スクリプト化する、自力でやる、の三択で考えるのがよさそうです。

## 8. batch は効いた

ここまでで、単発委譲には固定費があることが分かりました。

では、この固定費を減らせないか。

調べてみると、固定費の大半は、委譲1回ごとではなく、セッションを1本起動すること自体にかかっていました。

単発委譲を、セッション起動費 S と、1件あたりの作業費 T に分けます。

3件を単発で委譲した合計は $0.397。  
同じ3件を1セッションでまとめて委譲した合計は $0.192。

ここから逆算すると、S はだいたい $0.10、T はだいたい $0.03 になります。

つまり、起動費が大きいです。

なら、独立した委譲向きタスクが複数あるときは、1件ずつ投げるのではなく、まとめて投げればよい。

実測でも、3件をまとめた batch-trio では、1件あたり $0.064 まで下がりました。  
単発では負ける軽タスクでも、複数まとめると黒字化します。

この結果を受けて、手組みのバッチではなく、一級コマンドとして `lh batch` を作りました。

JSON マニフェストで複数タスクを渡します。  
各タスクには id、kind、check、prompt を持たせます。

タスクごとに check と修復ループを走らせます。  
失敗しても、独立タスクなら次へ進みます。  
最後に再検証スイープをして、後続タスクが前のタスクの成果を壊していないかも確認します。

さらに、タスクが終わるたびにセッションへ保存します。  
なので、呼び出し元の Claude Code が途中で落ちても、Local 側で完了した仕事は残ります。

実際、測定中に Claude Code 側が間欠的なネットワーク断で落ちることがありました。  
手組み委譲だと途中で止まりましたが、`lh batch` では委譲済みのタスクが残っていました。

この意味でも、batch はかなり効きました。

コストだけでなく、長い委譲作業の耐障害性にも効きます。

## 9. submit/wait はあまり効かなかった

一方で、うまくいかなかった機能もあります。

`lh submit / wait / poll` という非同期モードです。

狙いは、Local LLM が作業している間に、Claude Code が別の作業を進めることでした。

うまくいけば、壁時計時間を短くできるはずです。

まず、単一タスクの rename-sweep で試しました。

同期版は $0.110、521秒。  
非同期版は $0.147、539秒。

高くなって、速くもなっていません。

トランスクリプトを見ると、submit の3秒後に wait を呼んでいました。  
つまり、ほぼ即 wait しています。

これは考えると自然です。

作業指示書を書くには、事前に対象ファイルを読む必要があります。  
submit した時点で、待ち時間に回せる作業がもう残っていませんでした。

次に、本来の使い方も試しました。

大きいタスク A を Local に投げて、その間に Claude Code が軽い作業 B を自分でやる構成です。

overlap 自体は起きました。  
A が終わった後に残っている作業は、同期より短くなりました。

ただし、全体の壁時計は縮みませんでした。

理由は単純で、B が A よりずっと小さいからです。  
B を先に片付けても、結局 A を wait で待ちます。

短縮できる上限は B の数十秒程度で、Local 推論のばらつきに飲まれてしまいました。

なので、headless の自動委譲では、submit/wait は原則使わない、という結論になりました。

価値があるとすれば、人間が対話セッションで、Local に投げた作業とは別の同規模タスクを並行して進める場合だと思います。

## 10. 測定器が途中で変わった

今回の実験で、もうひとつ大きな学びがありました。

LLM エージェントのコスト測定は、かなり壊れやすいです。

途中で Claude Code のバージョンが 2.1.77 から 2.1.202 に上がりました。

その結果、システムプレフィックスのトークン数が大きく増えました。  
キャッシュ書き込みの実効単価も変わりました。

すると、同じような実験でも、昨日の $0.19 と今日の $0.53 が比較できなくなります。

cold / warm cache の差も大きいです。  
ターン1だけで $0.14 くらいずれることがありました。

なので、コストを測るときは、同日、同バージョン、warm 統制で比較しないとかなり危ないです。

これは、委譲そのものとは別に、今回かなり実用的な教訓でした。

LLM エージェントのドル実測は、思っているより測定器側の状態に依存します。

## 11. 実運用ではどう判断するか

最後に、実運用でどう判断するかを整理します。

まず、委譲してよさそうなのは、多ターンで機械的な作業です。

複数ファイルのリネーム。  
大きめのボイラープレート。  
まとまったテスト作成。  
ただし、受け入れコマンドを `--check` に渡せることが条件です。

軽いタスクが複数あるなら、1件ずつ `lh -p` で投げるのではなく、`lh batch` に束ねます。

次に、先にスクリプト化を疑うべきものがあります。

規則が明確に書ける作業。  
sed や Python や codemod で畳める作業。  
ファイル数は多いが、変換規則が単純な作業。

こういうものは、Claude Code 自身がスクリプトを書いて処理した方が速くて安いことがあります。

最後に、自力でやるべきもの。

数ターンで終わる軽作業。  
1行の docs 修正。  
型エラー2、3個の修正。  
設計判断が必要な作業。  
セキュリティや権限が絡む作業。  
急ぎの作業。

委譲は壁時計では遅くなります。  
だいたい3倍から7倍くらい遅くなる取引だと思った方がよいです。

また、委譲前には `lh stats --by-kind` を見るようにしました。

その種類のタスクで過去に失敗が多いなら、委譲しない。  
自力でやるか、作業指示を具体化してから再挑戦する。

feedback を記録するだけでなく、次の委譲判断に使う、という形です。

## 12. まとめ

今回のテーマは、Claude Code から Local LLM にコーディングタスクを委譲すると、トークンを節約できるのか、でした。

結論としては、条件付きでできます。

多ターンの機械的作業で、厳密に検証可能で、ワーカーが Local LLM で、受け入れコマンドを `--check` に渡し、軽いタスクは `lh batch` に束ねる。

この条件なら、実測で22%のコスト削減、6/6 PASS まで確認できました。

一方で、単発の軽作業は固定費に負けます。  
重い機械作業でも、Sonnet がスクリプト化して安く済ませることがあります。  
有料ワーカーへの委譲は二重払いになります。  
submit/wait は headless では壁時計を縮めませんでした。

今回一番印象が変わったのは、Local LLM 委譲を「モデルが安く働くから得」と見るのは少し雑だということです。

実際には、呼び出し元の固定費をどう償却するかが効いていました。

単発では固定費に負ける。  
複数なら batch で割れる。  
重すぎる機械作業は、そもそもスクリプト化できる。

なので、実運用では、委譲、スクリプト化、自力、の三択で考えるのがよさそうです。

Local LLM は、サブエージェントとして使えます。  
ただし、雑に投げるのではなく、検証と固定費を含めてワークフローを設計する必要があります。

残る宿題としては、`--resume` の実効性を check 失敗の実例で測ること、Local 推論の壁時計ばらつきを切り分けること、`lh stats --by-kind` の事前判断がどれくらい効くかを定量化することです。
