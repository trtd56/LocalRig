# Claude Code から Local LLM に仕事を委譲すると、本当にトークンは節約できるのか

> ドラフト v4(2026-07-07)。数値はすべて実測です。ただし各セル n=1 なので、統計的な比較というより、傾向を見るための検証として読んでください。ドル絶対額は測定時の Claude Code バージョンとキャッシュ状態にかなり依存します。

こんにちは

AIチームの戸田です

最近、手元の Mac で動く Local LLM を、Claude Code からサブエージェントのように使う実験をしています。

やりたいことはシンプルです。

Claude Code に全部やらせるのではなく、機械的で検証しやすい作業だけ Local LLM に投げる。そうすれば、Claude Code 側のトークン、つまり API コストを減らせるのではないか。

この発想自体は自然だと思います。一方で、実際に測ってみると「Local に投げれば安くなる」という単純な話ではありませんでした。

本記事では、Claude Code から Local LLM にコーディングタスクを委譲したとき、どれくらいコストが下がるのか、どこで品質が落ちるのか、どの条件なら実用になりそうかを検証した結果を共有します。

結論から言うと、**条件付きで節約できます**。ただし、単発委譲には固定費があり、軽いタスクでは逆に高くなります。さらに、品質面では Local LLM の能力よりも「検証をどこで行うか」の方が重要でした。

## 何を作ったか

手元で LocalRig というコーディングエージェントハーネスを作っています。Ollama 上の Qwen3.6 27B を、Claude Code のようなコーディングエージェントとして使うための TypeScript/Bun 実装です。

弱めのモデルでも安定して動くように、ツールコール修復、edit の多段マッチング、thinking 暴走の watchdog、コンテキスト管理などを入れています。

このハーネスには `lh` という CLI があります。Claude Code からは次のように呼べます。

```bash
lh -p - --json --cwd /path/to/repo --kind rename --check "bun test" <<'EOF'
<作業指示>
EOF
```

狙いは、Claude Code が「実装そのもの」を Local LLM に委譲し、Claude Code 側は依頼、検証、記録だけを担当する形にすることです。

では、それで本当に Claude Code 側のコストは下がるのでしょうか。

## 検証設定

比較したのは次の 3 条件です。

| 条件 | 内容 |
|---|---|
| baseline | Claude Code(Sonnet)が単独でタスクを解く |
| Local LLM 委譲 | Claude Code が `lh` 経由で Qwen3.6 27B に作業を委譲する |
| Haiku 委譲 | 委譲構造は同じで、委譲先だけ Haiku に差し替える |

タスクは、README 同期、偽バグ報告の再現不能判定、O(n^2) から O(n) への修正、12 ファイル横断リネーム、型エラー修復、ミューテーションテストを通るテスト作成などです。

主に見たのは次の3つです。

- `total_cost_usd`
- PASS / FAIL
- 実行時間

委譲したかどうかは、Claude Code の自己申告ではなく、`LH_HOME` にセッションファイルが生成されたかで判定しました。

## まず分かったこと

最初に分かったのは、委譲は万能なコスト削減ではないということです。

| タスク | baseline | 委譲 | 差 | 委譲発生 |
|---|---:|---:|---:|---|
| rename-sweep(12ファイル移行) | $0.337 | $0.126 | **-63%** | yes |
| perf-fix | $0.104 | $0.107 | ほぼ横ばい | yes |
| doc-sync(README同期) | $0.081 | $0.120 | **+49%** | yes |
| no-repro / type-repair / write-tests | - | - | ほぼ横ばい | no |

rename-sweep は大きく下がりました。一方で、doc-sync は逆に高くなっています。

これは、委譲には作業そのものとは別のコストがあるためです。Claude Code は Local LLM に投げる前に作業指示を書き、返ってきた結果を読み、差分を確認し、feedback を記録します。このオーケストレーション自体にトークンがかかります。

委譲したタスクの Claude Code 側コストを分解すると、だいたい次のような形でした。

```text
委譲時の Claude コスト ~= $0.11〜0.15
  - キャッシュ書込(セッション起動 + コンテキスト成長) ~$0.07
  - キャッシュ読出                                   ~$0.03
  - 出力(作業指示書・検証・feedback)                ~$0.02〜0.04
```

つまり委譲は、「実装コストをゼロにする」ものではありません。タスク規模に比例する実装コストを、固定のオーケストレーション費に変えるものです。

そのため、baseline が $0.15 くらいを超える作業では得をしやすく、数ターンで終わる軽い作業では負けやすくなります。

この時点で、Local LLM 委譲は「全部投げれば安い」ではなく、「固定費を超える作業だけ投げる」ものだと分かりました。

## 品質リスクはどこにあったか

次に、委譲をより強く促す system prompt に変えました。

具体的には、「最初の `lh` 呼び出しが返る前にファイルを編集しない」というレベルまで指示を強めました。すると委譲の遵守率は上がり、コストは baseline 比で **-23%** まで下がりました。

しかし、PASS は 6/6 から 5/6 に落ちました。

失敗したのは no-repro タスクです。これは、精巧な偽バグ報告に対して、コードを変更せず `verdict: not-reproducible` と報告できるかを見るタスクです。verify は TRIAGE.md の 1 行目の完全一致を要求していました。

Local Qwen の判断自体は正しかったです。ただし、TRIAGE.md の先頭に Markdown 見出しを置いてしまい、1 行目の完全一致条件を破りました。

問題はその後です。Claude Code は意味内容だけを見て、`feedback pass` を記録してしまいました。

ここで分かったのは、品質リスクの中心は「Local LLM が間違えること」だけではないということです。むしろ、**呼び出し元の検証が、受け入れゲートより浅くなること** が危険でした。

Claude Code が自力で同じタスクを解いた場合は PASS していました。つまり、委譲によって「作業者」と「検証者」が分かれたことで、新しい失敗経路が生まれたことになります。

## `--check` を入れる

この問題への対策として、`lh` に `--check` を入れました。

`--check` は、Local LLM の作業が終わった後に、`lh` 自身が受け入れコマンドを実行する仕組みです。失敗した場合は、Local LLM 側で修復リトライしてから結果を返します。

呼び出し元の Claude Code は、返ってきた JSON の `check.exit_code` と diff を確認してから feedback を記録します。

この形で再実測すると、結果は **6/6 PASS、コスト $0.817(baseline 比 -22%)** でした。

興味深いことに、修復ループは一度も発火していません。つまり、効いたのは自動修復ではありませんでした。受け入れゲートを、作業指示の中の曖昧な要件から、Local 側で実行される機械検証に移したことが効いていました。

`--check` はコストを下げる機能ではありません。品質を落とさないための機能です。

## Haiku に委譲するとどうなるか

ここまで見ると、気になるのは「これは自作ハーネスや Qwen の問題なのか」という点です。

そこで、委譲構造は同じまま、委譲先だけ Haiku に差し替えた対照実験も行いました。

| | baseline | lh 委譲(Qwen) | Haiku 委譲 |
|---|---:|---:|---:|
| PASS | 6/6 | 5/6 | 6/6 |
| 課金合計 | $1.053 | **$0.813(-23%)** | $1.225(**+16%**) |
| うちオーケストレーター | $1.053 | $0.813 | $0.907 |
| うちワーカー | - | $0(ローカル) | $0.318 |
| 実時間 | 372s | 1369s | 569s |

Haiku は 6/6 PASS でした。ただし、コストは baseline より **+16%** 高くなりました。

これは分かりやすい結果です。委譲には Claude Code 側の固定オーケストレーション費があります。そこに有料ワーカーの課金が乗ると、二重払いになります。

つまり、「委譲でコスト削減」が成立するのは、ワーカーの限界費用がほぼゼロのときだけです。Local LLM は、この構図で黒字を出せる数少ない構成でした。

## 重いタスクなら大きく勝てるのか

次に、もっと重いタスクなら委譲が大きく勝つのではないかと考えました。

用意したのは mass-migration という fixture です。40 ファイル・46 箇所にまたがる API 移行で、`log.write(level, msg)` を `logger.emit({level, msg, source})` に書き換えます。

`source` の正解値は各ファイル先頭の `// module-source:` コメントにだけ書かれており、パスからは導出できません。単純な置換だけでは解けないようにしています。

rename-sweep(12 ファイル $0.337)から考えると、baseline は $1.1〜1.5 くらいになると予想していました。

実測は **$0.23・88 秒・11 ターン** でした。

かなり外れました。

理由はトランスクリプトを見るとすぐ分かりました。Sonnet は 46 箇所を手で編集していません。コメントから正解値を機械抽出する Python スクリプトを書き、一括で書き換えていました。

つまり、baseline はファイル数に線形で増えません。規則が書ける機械的タスクであれば、Sonnet は自分でスクリプト化して安く済ませます。

委譲アームは **PASS・$0.129・599 秒** でした。節約は **-44%** なので十分意味はあります。ただ、当初期待していた「-80%」のような世界ではありませんでした。

ここで見えたのは、次の構造です。

Local LLM に安心して委譲できるほど機械的なタスクは、Sonnet がスクリプトで畳めるほど機械的なタスクでもあります。

そのため、委譲には下限と上限があります。下には固定費の壁があります。上にはスクリプト化の天井があります。

実運用では、「委譲するか、自力でやるか」ではなく、**委譲する / スクリプト化する / 自力でやる** の三択で考えるのがよさそうです。

## batch は効いた

ここまでで、単発委譲には $0.11〜0.18 程度の固定費があると分かりました。

ただし、その固定費の中身をもう少し見ると、重要なことが分かりました。固定費の大半は、委譲 1 回ごとではなく、セッションを 1 本起動すること自体にかかっていました。

単発委譲を「セッション起動費 S + 1件あたりの作業費 T」と置くとします。3 件を単発で委譲した合計は $0.397。同じ 3 件を 1 セッションでまとめて委譲した合計は $0.192 でした。

ここから逆算すると、**S ~= $0.10、T ~= $0.03** になります。

`batch-trio` fixture での実測は次の通りです。

| 条件 | 判定 | Claude コスト | 1件あたり |
|---|---|---:|---:|
| baseline(Sonnet 単独) | 3/3 PASS | $0.351 | $0.117 |
| 単発委譲 3件合計 | 3/3 PASS | $0.397 | $0.132 |
| バッチ委譲(1セッション) | **3/3 PASS** | **$0.192** | **$0.064** |

単発では負ける軽タスクでも、複数まとめると黒字化します。

この結果を受けて、手組みのバッチではなく、一級コマンドとして `lh batch` を作りました。JSON マニフェストで独立タスクをまとめて渡し、タスクごとの check、修復、部分失敗の続行、feedback、最後の再検証スイープまでを 1 コールに畳む形です。

ここで予想外だったのは、実装そのものよりも測定環境でした。

第5ラウンドの測定と第6ラウンドの測定の間に、Claude Code が 2.1.77 から 2.1.202 に自動更新されていました。システムプレフィックスは 10,927 から 35,592 トークンへ増え、キャッシュ書き込みの実効単価も変わっていました。

同日に 3-way を取り直すと、こうなりました。

| 条件 | 判定 | フル推定(warm) | baseline 比 |
|---|---|---:|---:|
| baseline(Sonnet 単独) | PASS | $0.446 | - |
| 手組み(3x `lh -p`) | PASS | 約 $0.60+ | +27%以上の赤字 |
| 一級 `lh batch`(1コール) | PASS | 約 $0.39〜0.42 | -6〜13% の黒字 |

新しい会計では、手組みで `lh -p` を何度も呼ぶ形は baseline に負けました。1 コールに畳む `lh batch` だけが黒字に残りました。

また、`lh batch` には副次的なメリットもありました。測定当日は間欠的なネットワーク断で、呼び出し元の Claude Code が何度か落ちました。しかし `lh batch` はマニフェスト受理後の実行、check、保存が Local 側で完結します。呼び出し元が落ちても、委譲済みの仕事は残りました。

コストだけでなく、長い委譲作業の耐障害性という意味でも、batch 化は効きました。

## submit/wait はあまり効かなかった

もう一つ、`lh submit / wait / poll` という非同期モードも作りました。

狙いは、Local LLM が作業している間に Claude Code が別の作業を進めることです。うまくいけば、壁時計時間を短くできるはずです。

まず単一タスク(rename-sweep)で測ると、同期版は $0.110・521 秒、非同期版は **$0.147・539 秒** でした。submit の 3 秒後に wait を呼び、約 8.3 分ブロックしていました。

これは考えてみると自然です。作業指示書を書くには、事前に対象ファイルを読んでおく必要があります。submit した時点で、待ち時間に回せる作業が残っていませんでした。

次に、本来の使い方も試しました。大きいタスク A を委譲し、その間に Claude Code が軽い作業 B を自分でやる形です。

overlap 自体は起きました。A 終了後に残ったクリティカルパスは、同期 40 秒から非同期 16 秒へ縮みました。

ただし、全体の壁時計は縮みませんでした。非同期 424 秒、同期 384 秒でした。

理由は、B が A よりかなり小さいからです。B を先に片付けても、結局 A を `lh wait` で待ちます。短縮できる上限は B の 24 秒程度で、Local 推論のばらつきに飲まれてしまいました。

そのため、headless の自動委譲では `submit/wait` は原則使わない、という結論です。

価値が残るとすれば、人間が対話セッションで、Local に投げた作業とは別の同規模タスクを並行して進める場合だと思います。

## 実運用ではどう判断するか

今回の実測から見ると、判断基準は次のようになりそうです。

### 委譲する

- 多ターンの機械的作業
- 複数ファイルのリネーム、ボイラープレート、大きめのテスト作成
- ただし、1 本のスクリプトには畳めないもの
- 受け入れコマンドを `--check` に渡せるもの
- 軽いタスクが複数ある場合は `lh batch` に束ねられるもの

### 先にスクリプト化を考える

- 規則が明確に書ける作業
- sed / Python / codemod で畳める作業
- ファイル数は多いが、変換規則が単純な作業

### 自力でやる

- 数ターンで終わる軽作業
- 1 行の docs 修正
- 型エラー 2〜3 個の修正
- 設計判断が必要な作業
- セキュリティや権限が絡む作業
- 急ぎの作業

また、委譲前には `lh stats --by-kind` を見るようにしました。その種類のタスクで失敗が多いなら、委譲せず自力でやるか、作業指示を具体化してから再挑戦します。

設定にも注意が必要でした。

「常に委譲せよ」という緩い指示は、半分くらい無視されました。しかも、無視されたタスクは損益分岐近くのものが多く、結果的には合理的な不服従でした。

コスト以外の理由、たとえばプライバシーやオフライン要件で必ず Local に投げたい場合は、「最初の `lh` 呼び出しが返る前にファイルを編集しない」くらい明示した方がよさそうです。

## 測定で気をつけること

今回、かなり強く感じたのは、LLM エージェントのコスト測定は壊れやすいということです。

特に Claude Code の自動更新は大きかったです。2.1.77 から 2.1.202 で、システムプレフィックスとキャッシュ単価が変わりました。同じ実験でも、昨日の $0.19 と今日の $0.53 はそのまま比較できません。

cold / warm cache の差も大きいです。ターン 1 だけで約 $0.14 ずれることがありました。

また、prompt token 数の合計を見てもコストとはあまり対応しません。`input + cache_read + cache_creation` を足しても、cache read は単価が違うため、コスト比較としては分かりにくいです。比較するなら `total_cost_usd` を見る必要があります。

今回の数値はすべて n=1 です。そのため、統計的な有意差を主張するものではありません。効果が大きく出る設計にして、方向性を見るための検証としています。

## まとめ

Claude Code から Local LLM にコーディングタスクを委譲すると、トークンを節約できるのか。

今回の検証では、**条件付きでできます**。

多ターンの機械的作業で、厳密に検証可能で、ワーカーが Local LLM で、受け入れコマンドを `--check` に渡し、軽いタスクは `lh batch` に束ねる。この条件なら、実測で **-22%・品質退行ゼロ** まで確認できました。

一方で、単発の軽作業は固定費に負けます。重い機械作業でも、Sonnet がスクリプト化して安く済ませることがあります。有料ワーカーへの委譲は二重払いになります。`submit/wait` は headless では壁時計を縮めませんでした。

今回の検証で一番印象が変わったのは、委譲を「Local LLM が安く働くから得」と見るのは少し雑だということです。

実際には、呼び出し元の固定費をどう償却するかが効いていました。単発では固定費に負ける。複数なら batch で割れる。重すぎる機械作業は、そもそもスクリプト化できる。

そのため、実運用では **委譲 / スクリプト化 / 自力** の三択で考えるのがよさそうです。

残る宿題は、`--resume` の実効性を check 失敗の実例で測ること、Local 推論の壁時計ばらつきを切り分けること、`lh stats --by-kind` による事前判断の効果を定量化することです。

公開前には、主要な数値を同日・同バージョン・warm 統制で複数回取り直したいと思っています。
