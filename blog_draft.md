# Claude Code からローカル LLM に仕事を委譲すると、本当にトークンは節約できるのか — 実測してわかった「固定費の壁」と品質を落とさない委譲の作り方

> ドラフト v2(2026-07-07)。数値はすべて実測(各セル n=1)。ドル絶対額は測定時の Claude Code バージョンの会計に依存する(第6ラウンド参照)。社名・製品名等の体裁は公開媒体に合わせて調整してください。

## TL;DR

- Claude Code(Sonnet)からローカル LLM(Qwen3.6 27B / Ollama)へコーディングタスクを委譲する CLI を作り、**委譲で Claude 側の API コストが本当に下がるのか**を 5 ラウンド+対照実験で実測した
- 結論: **下がる。ただし条件付き**。最終形で **−22%、品質退行ゼロ(6/6 PASS)**。ただし委譲には**タスク規模によらない固定オーケストレーション費(~$0.11〜0.18)**があり、損益分岐(baseline ≈ $0.15)未満の軽いタスクでは単発だと**逆に割高**になる。その固定費の大半は**セッション起動費**で件数に依存しないと後で判明し、**独立タスクを1セッションにまとめて委譲すると 1件あたり $0.064(単発比 −52%)** まで下がる——軽タスクもバッチなら黒字化する
- 予告していた 2 つの追試も決着した: **バッチ委譲は当たり(1件 $0.064、単発比 −52%)、非同期 submit/wait は外れ(headless では壁時計を縮められず、原則使わない)**。委譲判断は事前に `lh stats --by-kind` の実績(その種類の合格率)を見て較正する
- 委譲が最も効くはずの重量級・機械的タスクは、**baseline 側(Sonnet)が Python スクリプトに畳んで安く済ませてしまう**(予測 $1.1〜1.5 の 40 ファイル移行が実測 $0.23)。委譲は下限の「固定費の壁」と上限の「スクリプト化の天井」に挟まれ、現実的な節約は **−30〜50%** が上限だった。判断は「委譲 vs 自力」の二択ではなく **「委譲 / スクリプト化 / 自力」の三択**で考えるのが正しい
- この固定費は委譲先を替えても消えない**呼び出し元(Claude Code)側の構造費**だった。委譲先を Haiku(有料 API)に替える対照実験では**むしろ +16% の純損**——「委譲でコスト削減」が成立するのは**ワーカーの限界費用がほぼゼロ(=ローカル)のときだけ**
- 最大の品質リスクはローカルモデルの能力ではなく、**「呼び出し元の検証が受け入れゲートより浅くなる」偽 pass** だった。受け入れコマンドを委譲先が自分で実行する `--check` 機構で構造的に封鎖できた
- 第6ラウンドでバッチ委譲を一級コマンド `lh batch` に昇格させて再実測したら、**測定対象より先に物差し(Claude Code の会計)が変わっていた**: CLI 更新でシステムプレフィックスが約3.3倍・キャッシュ書き単価が1.6倍になり、**「N 回の `lh -p` を手で並べる」旧来の委譲はベースラインに負ける赤字に転落**。ターン数を1コールに畳む `lh batch` だけが黒字を維持した(対手組み ≈−30%)。**ドルで語る実測はすべて「同日・同バージョン・キャッシュ warm 統制」でしか比較できない**——これ自体が一連の実験で最も汎用性の高い教訓かもしれない

## 背景: LocalRig と `lh` 委譲

手元の Mac で動くコーディングエージェントハーネス LocalRig を開発している。Ollama 上の Qwen3.6 27B(量子化 16GB)を Claude Code 級のコーディングエージェントにするための TypeScript/Bun 実装で、弱いモデル向けの補強(ツールコール修復、edit の多段マッチング、thinking 暴走の watchdog、コンテキスト管理など)を積んでいる。20 タスクの自動評価スイートで Claude Code(Sonnet)と同じタスクを解かせ、20/20 PASS まで来ていた。

このハーネスには `lh` という CLI があり、Claude Code から「サブエージェント」として呼べる:

```bash
lh -p - --json --cwd /path/to/repo --kind rename --check "bun test" <<'EOF'
<作業指示>
EOF
# → {"session_id":"...","status":"ok","check":{"exit_code":0,...},"report":{...},...}
```

設計上の狙いは「機械的で検証可能な作業をローカルに逃がして Claude のトークン(=課金)を節約する」こと。**ではその節約は、実際いくらなのか?** 誰も測っていなかったので測った。

## 実験設計

比較は 3 アーム。ポイントは「同じタスク・同じプロンプト」で条件だけを変えること。

| アーム | 説明 |
|---|---|
| baseline | Claude Code(Sonnet)が単独でタスクを解く |
| claude-delegate | 同じ Claude Code に、system prompt 追記(`--append-system-prompt`)で「実装は `lh` に委譲せよ」と指示。**タスク文は 1 バイトも変えない** |
| claude-delegate-haiku | 委譲構造は同一、委譲先だけ `claude -p --model haiku`(有料 API ワーカー)に差し替えた対照群 |

- 対象は委譲ユースケース向けに設計した 6 タスク(README 同期 / 偽バグ報告の再現不能判定 / O(n²)→O(n) 修正 / 12 ファイル横断リネーム / 型エラー修復 / ミューテーションテストを通るテスト作成)
- 主要メトリクスは `total_cost_usd`(キャッシュ料金体系を織り込んだ、Anthropic が実際に請求する額)
- **委譲の発生は自己申告に頼らず機械判定**: 実行ごとに `LH_HOME` を隔離ディレクトリに向け、そこにセッションファイルが生成されたかで判定。Haiku アームはワーカーの結果 JSON ファイル+ Claude Code のセッショントランスクリプト数で二重に判定
- 判定は各タスクの `verify.sh`(テスト+grep+sha256 改ざん検出)

## 第1ラウンド: 「委譲すれば安くなる」は半分ウソだった

| タスク | baseline | 委譲 | 差 | 委譲発生 |
|---|---|---|---|---|
| rename-sweep(12ファイル移行) | $0.337 | $0.126 | **−63%** | ✅ |
| perf-fix | $0.104 | $0.107 | ±0 | ✅ |
| doc-sync(README同期) | $0.081 | $0.120 | **+49%** | ✅ |
| no-repro / type-repair / write-tests | — | — | ほぼ横ばい | ❌ 自力実行 |

2 つの想定外があった。

**その 1: 軽いタスクは委譲すると高くなる。** doc-sync は baseline 29 秒・$0.081 の軽作業だが、委譲すると $0.120(+49%)。作業指示書を書き、ローカルの完了を待ち、結果を検証し、feedback を記録する——このオーケストレーション作業自体が Claude のトークンを食う。

**その 2: 「必ず委譲せよ」と system prompt で命じても、Sonnet は半分しか従わない。** 6 タスク中 3 つで Claude は `lh` を一度も呼ばず自力実行した(トランスクリプトで確認)。しかも無視されたタスクはどれも損益分岐点近傍で、**economically rational な不服従**だった。

## 発見: 委譲は「変動費の固定費化」である

委譲したタスクの Claude 側コストを分解すると、タスク規模によらずほぼ一定だった:

```
委譲時の Claude コスト ≈ $0.11〜0.15
  ├─ キャッシュ書込(セッション起動+コンテキスト成長) ~$0.07
  ├─ キャッシュ読出                                   ~$0.03
  └─ 出力(作業指示書・検証・feedback)                ~$0.02〜0.04
```

つまり委譲とは「タスク規模に比例する実装コスト」を「規模非依存の固定オーケストレーション費」に変換する操作で、**節約額 = baseline − 固定費**。baseline が固定費を上回るタスク(多ターンの機械作業)でだけ得をする。33 ターンかかっていた rename-sweep が −63% で大勝ちし、6 ターンで終わる doc-sync が +49% で負けたのはこの構造の必然だった。

さらに重要なのは、固定費の大半(~$0.07〜0.10)が **Claude Code が起動して会話を進めるだけで発生する費用**だという点。これは baseline 側にも同額含まれる。doc-sync の baseline $0.081 のうち $0.051 は起動費で、実装作業ぶんは $0.03 しかない——**完璧な委譲先が無料で瞬時に働いても、絶対に勝てない**タスクだったのだ。

## 第2ラウンド: 強制委譲で見えた本当の品質リスク

nudge を強化(「最初の `lh` 呼び出しが返る前にファイルを編集するな」)すると遵守は 6/6 になり、コストは baseline 比 **−23%**。ところが PASS が 5/6 に落ちた。

落ちた no-repro タスクの解剖が今回いちばんの収穫だった。これは「精巧な偽バグ報告に対し、コードを変更せず `verdict: not-reproducible` と正直に報告できるか」を試すタスクで、**1 行目が完全一致であること**を verify が要求する。

- ローカル Qwen は判断は正しかった。しかし TRIAGE.md の先頭に Markdown 見出しを置き、厳密な 1 行目要件を破った
- そして **Claude は意味内容だけ検証して `feedback pass` を記録した**。受け入れゲート(1 行目の完全一致)より検証が浅かった

つまり委譲の品質リスクの正体は「ローカルモデルがバカ」ではなく、**「検証者の検証が受け入れゲートより浅くなる」**こと。同じタスクを Claude が自力でやったラウンドでは PASS していたので、これは委譲によって新たに混入した故障モードである。

## Haiku 対照実験: 「ツールが悪いのか、ローカル LLM が悪いのか」

ここまでの結果を見て湧く疑問——この厳しさは自作ハーネスの問題なのか、それともローカル LLM 委譲がそもそも筋悪なのか。切り分けるため、**委譲構造は同一のまま委譲先だけ Haiku(有料 API)に替えた**対照実験をやった。

| | baseline | lh 委譲(Qwen) | Haiku 委譲 |
|---|---|---|---|
| PASS | 6/6 | 5/6 | 6/6 |
| 課金合計 | $1.053 | **$0.813(−23%)** | $1.225(**+16%**) |
| うちオーケストレーター | $1.053 | $0.813 | $0.907 |
| うちワーカー | — | $0(ローカル) | $0.318 |
| 実時間 | 372s | 1369s | 569s |

3 つの仮説が一度に決着した:

1. **固定費は委譲先非依存**(lh 版 $0.813 vs Haiku 版オーケストレーター分 $0.907、doc-sync 単体では $0.120 vs $0.118 でほぼ一致)。floor は呼び出し元 Claude Code 側の構造費であり、**ハーネスを Pi や OpenCode に替えても変わらない**
2. **ワーカー品質の差は小さい**: Haiku 6/6 vs Qwen 5/6。差は「厳密フォーマット遵守 1 件」だけ
3. **実時間 3.7 倍のうち委譲構造は ~1.5 倍分だけ**。残りはローカル推論速度(ハードウェア)
4. おまけに決定的な発見: **有料ワーカーへの委譲は floor+ワーカー課金の二重払いで純損(+16%)**。「委譲でコスト削減」はワーカーの限界費用がゼロのときだけ成立する。ローカル LLM はこの構図で唯一黒字を出せる構成だった

## 改良: 検証を「正しい場所」に移す

品質リスクの正体が「検証の位置」だと分かれば、対策は明確だった。委譲先の CLI に実装したのは:

- **`--check "<受け入れコマンド>"`**: エージェントループ完了後、lh 自身が受け入れコマンドを実行。失敗ならローカルモデル(タダ)で修復リトライしてから返す。結果は `check.exit_code / attempts / output_tail` として JSON に含まれる
- **`report.changed_files / commands_run`**: 呼び出し元が `git diff` や再 read に使うターンを削る機械可読レポート
- **`--kind` タグ + `lh stats --by-kind`**: 「この種類のタスクは委譲で成功しているか」を実データで判断する学習ループ
- **`lh submit / wait / poll`**: 呼び出し元が最大 20 分ブロックされる問題への非同期モード

### 第3ラウンド: 品質退行ゼロの委譲

`--check` 前提の指示で再実行した結果:

| | baseline | 第2R(強制) | **第3R(--check)** |
|---|---|---|---|
| PASS | 6/6 | 5/6 | **6/6** |
| コスト | $1.053 | $0.813 | **$0.817(−22%)** |

- **no-repro が PASS に回復**。全 6 タスクで check は exit 0・attempts 1——興味深いことに**修復ループは一度も発火していない**。受け入れゲートが「作業指示に埋め込まれた --check コマンド+ローカル側での機械検証」に移ったことで、偽 pass の経路が構造的に消えた(修復が効いたのではなく、**検証の位置が正しくなった**)
- コストは横ばい($0.813→$0.817)。固定費 floor は --check では動かない——事前予測どおりで、**--check が買ったのはコストではなく品質**

## 第4ラウンド: 「重量級なら −80%」の予測は、面白い形で外れた

前回の「今後」で予告した重量級タスクの実測をやった。結論から言うと、予測は外れた——しかも構造的に面白い外れ方だった。

用意した新 fixture は **mass-migration**: 40 ファイル・46 箇所にまたがる API 移行で、`log.write(level, msg)` を `logger.emit({level, msg, source})` に書き換える。ミソは `source` の正解値で、これは各ファイル先頭の `// module-source:` コメントにだけ書かれており、**パスからは導出できない**。単一の sed では解けないよう設計した「自明ではないが機械的」なタスクだ。

rename-sweep(12 ファイル $0.337)からの線形外挿で、baseline(Sonnet 単独)は **$1.1〜1.5** かかると予測していた。実測は——**$0.23・88 秒・11 ターン**。予測の 5〜6 分の 1 である。

トランスクリプトを見て理由がわかった。Sonnet は 46 箇所を手で編集していなかった。**コメントから正解値を機械抽出する Python スクリプトを書き、46 箇所を一括で書き換えていた**。つまり baseline はファイル数に線形でスケールしない——規則が記述できる機械的タスクを、Sonnet はスクリプトに畳んで償却する。12 ファイル $0.337 → 40 ファイル $0.23 と、ファイル数 3.3 倍でむしろ安くなった(rename-sweep はファイルごとの手編集で 33 ターンかかったが、mass-migration はスクリプト化が刺さって 11 ターンで終わった)。

委譲アーム(`--check` つき)は **PASS・$0.129・599 秒**。ローカル Qwen も bash ループ + sed で概ねスクリプト的に解いた(18 ターン・555 秒・check exit 0・修復ループ発火ゼロ・feedback pass)。節約は **−44%**($0.23→$0.129)で、固定費 floor($0.11〜0.15 帯)は 40 ファイル規模でも動かないことを再確認した。だが**予告していた「−80% 級」は幻だった**。

### 「固定費の壁」の向かいには「スクリプト化の天井」がある

この結果には、これまでの固定費の話と対になる構造的な教訓がある。

委譲のスイートスポットは「機械的+厳密検証可能」なタスクだ。ローカル 27B に安心して任せられるのは、まさにそういうタスクだから。ところが——**「ローカルに安心して委譲できるほど機械的なタスク」は、「Sonnet がスクリプト一発で畳めるほど機械的なタスク」でもある**。委譲が得意な領域と、スクリプト化が得意な領域は、ほぼ完全に重なっている。

だから baseline は、委譲が最も効くはずの機械的タスクでこそ、スクリプト化して自分で安く済ませてしまう。「baseline $1 超の純粋に機械的なタスク」という委譲の理想的な狙い所は、**構造的にほぼ空集合**なのだ。もしそんなタスクが存在するなら Sonnet もスクリプトで畳めるので、baseline がそもそも $1 に届かない。

現実的な節約は「floor(≈$0.13)vs baseline $0.2〜0.4」の **−30〜50% 帯が上限**、というのが 4 ラウンドを通した結論になった。委譲は「固定費の壁」(下限=軽いタスクでは損)と「スクリプト化の天井」(上限=重いタスクは Sonnet が自分でスクリプト化)に挟まれた、意外に狭い帯の中でだけ得をする道具だった。運用の頭の切り替えとしては、「委譲するか、自分でやるか」の二択ではなく、**「委譲するか、スクリプト化するか、自分でやるか」の三択**で考えるのが正しい。

## バッチ委譲: 固定費は「委譲1回あたり」ではなく「セッション1本あたり」だった

第5ラウンドで、固定費モデルにもう一段の解像度が加わった。ここまで「委譲は ~$0.11〜0.18 の固定費を課す」と書いてきたが、この固定費を分解すると内訳が見えた。単発委譲を「セッション起動費 S + 1件あたりの作業費 T」と置き、3件を単発で委譲した合計($0.397 = 3×(S+T))と、同じ3件を1セッションでまとめて委譲した合計($0.192 = S+3T)を連立させると、**S ≈ $0.10・T ≈ $0.03** が出る。つまり固定費の大半(S)は**セッションを 1 本起動すること自体**の費用で、委譲件数には依存しない。「floor は委譲1回あたり」だと思っていたが、実は「セッション1本あたり」だったのだ。

含意は大きい。**独立した委譲向きタスクが複数あるなら、別々のセッションで単発委譲するのではなく、1セッションにまとめて次々委譲すればいい**。起動費 S が件数で割られ、1件あたり実効費は限界費 T($0.03)に漸近する。独立3件(docs / types / perf 級)を1リポジトリに同居させた `batch-trio` fixture で実測した:

| アーム | 判定 | Claude コスト | 1件あたり |
|---|---|---|---|
| baseline(Sonnet 単独) | 3/3 PASS | $0.351 | $0.117 |
| 単発委譲 3件合計 | 3/3 PASS | $0.397 | $0.132 |
| **バッチ委譲(1セッション)** | **3/3 PASS** | **$0.192** | **$0.064** |

1件あたり $0.064 は単発 floor($0.132)比 **−51.6%**、baseline 比でも −45.3%。壁時計もバッチ 536 秒は単発3件合計 568 秒より速い(起動が 1 回で済むため。なお 3 件はローカル側で逐次実行された)。

だから損益分岐は 1 本の線ではなく**2 段書き**になる: **単発委譲なら 1件 ≈ $0.11〜0.18(= S+T、損益分岐 baseline ≈ $0.15)、バッチ委譲なら 1件 ≈ $0.06〜0.08(実測 $0.064)**。単発では負ける doc-sync 級の軽タスク(+49% の負け)も、他の委譲と束ねて起動費を分担すれば黒字化する。運用の指針はシンプルだ——「委譲したい軽タスクが 1 件だけなら自分でやる、複数たまったらまとめて 1 セッションで委譲する」。

## 第6ラウンド: バッチを一級コマンド化したら、測定対象より先に「物差し」が変わった

第5ラウンドのバッチ償却は、Claude に「同一セッション内で `lh -p` を順に呼べ」と指示して束ねた、いわば手組みのバッチだった。これを一級サブコマンド `lh batch` に昇格させた——JSON マニフェストで独立タスクを一括投入し、タスク毎に check+修復ループ、部分失敗の続行、タスク単位の feedback、完了後の**再検証スイープ**(あるタスクの bash 副作用が先行タスクの成果を巻き戻していないか、通過済み check を最後に全部もう一度回す)までを 1 コールに畳む。実装後の初回評価は 35 分ハングし、診断で設計ミスを 3 つ修正した(`--max-time` をバッチ**総予算**に/タスク毎に 27B の文脈をフルリセット——文脈を積むと後半のタスクで確実に失速する/タスク完了毎の逐次保存)。

再実測すると、一級化した batch のコストが手組み($0.192)の 2.7 倍の $0.526 に見えた。「実装が悪いのか?」と transcript の生 usage を突き合わせて分かったのは、**測定対象ではなく物差しが変わっていた**ことだ。第5ラウンドの測定(7/6 昼)と第6ラウンドの測定(7/7)の間に Claude Code が 2.1.77 から 2.1.202 に自動更新され、(1) システムプレフィックスが 10,927→35,592 トークンに肥大(=全ターンにかかる文脈税が倍増)、(2) キャッシュ書き込みの実効単価が $3.75→$6/M に変化、(3) おまけに新測定だけプレフィックスキャッシュが cold だった(ターン1で約 $0.14 の差)。**ドルで語る実測は、同日・同バージョン・キャッシュ warm 統制でしか比較できない。**

そこで同日に 3-way を取り直した(全run 2.1.202・warm、warm 換算と欠落ターン補正は transcript 実測に基づく):

| アーム | 判定 | フル推定(warm) | baseline 比 |
|---|---|---|---|
| baseline(Sonnet 単独) | PASS | $0.446 | — |
| 手組み(3× `lh -p`) | PASS | **≈$0.60+** | **+27%以上の赤字** |
| **一級 `lh batch`(1コール)** | PASS | **≈$0.39〜0.42** | **−6〜13% の黒字** |

結果は皮肉だが示唆的だ。**新しい会計の下では、第5ラウンドで「当たり」だった手組みバッチ委譲がベースラインに負ける**。プレフィックス肥大でオーケストレータの 1 ターンが高くなったため、「lh を N 回呼んで N 回結果を処理する」ターン数そのものが罰金になった。1 コールに畳む `lh batch` だけが黒字に残る(対手組み ≈−30%)。つまり CLI の会計変更は、バッチ一級化の価値を**測定の途中で勝手に引き上げた**。

もうひとつ、一級化の価値はコストだけではなかった。測定当日は後述の間欠ネットワーク断で呼び出し元の Claude が 4 run 中 3 run 死んだのだが、`lh batch` はマニフェスト受理後の実行・check・修復・永続化がすべてローカル側で完結するため、**呼び出し元がいつ死んでも委譲済みの仕事は完遂されて残る**(手組みは 3 件中 1 件を委譲したところで死んで FAIL、バッチは 2 run とも 3/3 完遂)。逐次保存はハング対策として入れた修正だったが、結果的に最大の保険になった。

なお正直に書くと、ローカル側の壁時計は run 間の分散が凄まじい。同一 3 タスクのバッチをアーカイブ 6 run で比べると合計 333〜1688 秒(5 倍)、単一タスクでも 80→993 秒までぶれる(ターン数はほぼ同じで、1 ターンあたりの推論時間だけ伸びる——Ollama 側の状態を疑っている)。最速 run は手組み 3 連発と同等なので「一級 batch が構造的に遅い」わけではないが、**n=1 の壁時計比較は従来想定よりずっと信頼できない**ことがはっきりした。

## 非同期モード(submit/wait)は、本来の使い所でも元が取れず、「使わない機能」に決着した

もう一つの予告——`submit`/`wait` 非同期モードによる待ち時間回収——は 2 ラウンドかけて実測し、負の結果で決着した。

まず第4ラウンド、単一タスク(rename-sweep)を `submit`→`wait` で再実測すると **PASS・$0.147・539 秒**(同期版は $0.110・521 秒)。トランスクリプトに決定的な証跡があった: 検証準備の read は submit の**前**に完了しており、submit の**3 秒後**に wait を呼んで約 8.3 分ブロックしていた。作業指示書を書くにはどのみち事前に対象ファイルを読む必要があるので、**submit した時点で、待ち時間に回せる作業がもう残っていない**。非同期化はブロック時間を 1 秒も回収せず、ターン増でコストが **+33%** になっただけだった。

ただしこれは「単一タスクだから待ち時間に回す仕事がない」だけかもしれない。そこで第5ラウンドで**本来の使い所**を測った: 委譲向きの大タスク A(10 ファイル移行、ローカル ~5 分)と、Claude 自身が書くべき軽作業 B(設計メモ作成)を同居させ、同期版(A を待ち切ってから B)と非同期版(A を submit → B を自分で → wait)を比べた(`async-pair` fixture)。

結果は微妙だが決定的だった。**overlap の機構自体は確かに働いた**——A 終了後に残ったクリティカルパスは同期 40 秒(B+検証+feedback)から非同期 16 秒(検証+feedback のみ)に縮み、差の ~24 秒ぶん、B が A のバックグラウンド実行中に片付いていた。**にもかかわらず net 壁時計は縮まなかった**(非同期 424 秒 > 同期 384 秒)。理由は構造的だ: 設計前提が「大きい A を委譲し、小さい B を自分で持つ」である以上 **B ≪ A が保証される**。B(~24 秒)を早々に片付けても、残った A(~300 秒)を結局 `lh wait` でブロックする——同期のブロックと変わらない。理論上の短縮上限は min(A, B) = B ≈ 24 秒(総時間の約 6%)にすぎず、ローカル推論のばらつきだけで ±31 秒(8%)動くので、その中に飲まれてしまう。

コストは非同期が **−12%**($0.281→$0.247)だったが、これは latency の効果ではなく**キャッシュ会計の副作用**だ。B を A の裏で一続きに処理したためコンテキストを 1 回だけ構築でき、高価なキャッシュ生成が 12,314 トークン少なく済んだ(同期は「長時間ブロック → 再開して B」の分断でキャッシュ再生成が増える)。n=1 で再現性は未検証だし、壁時計短縮の対価ではない。

2 ラウンドの結論はシンプルだ: **headless(`-p`)の自動委譲では submit/wait を使わない**。単一タスクでも本来の A‖B でも net 壁時計は縮まず、コストはむしろ増える。価値が残るとすれば、人間が対話セッションで、ローカルに投げた作業とは無関係な同規模の別作業を並行に抱える場合だけ——これは headless 評価では原理的に測れない。

**その他の実装メモ:** 差し戻し時にフル作業指示を再構築せず同一文脈へ追撃指示を送れる `--resume <session_id>` を実装した(check 失敗の実例が出たら実効性を実測する)。`--worktree` パッチモード(並行編集を隔離する案)は見送り確定——`--check` の導入で当初の動機だった検証の位置ずれが解消し、並行編集の事故も一度も観測していない。

## 実運用ガイド: いつ委譲すべきか

実測から導いた運用指針。CLAUDE.md / AGENTS.md に委譲を設定する人向け。判断は「委譲するか、自分でやるか」の二択ではなく、**「委譲 / スクリプト化 / 自力」の三択**で考える。

**委譲する:**
- 多ターンの機械的作業(複数ファイルのリネーム・ボイラープレート・大きめのテスト作成)で、**1 本のスクリプトには畳めない**もの(rename-sweep は baseline が 33 ターン手編集した=スクリプト化できなかった好例)。目安は「自分でやると baseline $0.15〜0.4」。下は固定費の壁、上はスクリプト化の天井で、得をする帯は思ったより狭い
- 厳密に検証可能なタスクのみ。**受け入れコマンドを `--check` に渡し、`check.exit_code == 0` と diff を確認してから合格を記録**
- **委譲したい軽タスクが複数たまったら `lh batch` の 1 コールに束ねる**(手組みで `lh -p` を並べない)。固定費の大半はセッション起動費なので束ねれば 1 件あたりが下がり(2.1.77 会計の実測で $0.064、単発比 −52%)、さらに 2.1.202 会計では手組み逐次委譲が赤字化したため、**束ね方は一級コマンド一択**になった(同日実測: batch ≈$0.39〜0.42 vs 手組み ≈$0.60+ vs baseline $0.446)
- **委譲判断の前に `lh stats --by-kind` を見る**。その種類(`--kind`)の実績が graded ≥ 3 件あって合格率 50% 未満(fail が過半)なら、その種類は委譲せず自力実行するか、作業指示を具体化してから再挑戦する

**スクリプト化する(委譲より優先):**
- 規則が明確に記述でき、sed / Python 1 本で畳めるなら、委譲するより自分でスクリプトを書いて回す方が速くて安い。重量級でも baseline は $1 に届かず、委譲の固定費すら下回ることが多い(mass-migration の実測: 40 ファイル・46 箇所の API 移行が baseline $0.23)。「委譲に投げたい」と思うほど機械的なタスクは、まずスクリプト化を疑う

**委譲しない(自力):**
- 数ターンで終わる軽作業(1 行の doc 修正、型エラー 2〜3 個)。固定費 ~$0.11〜0.18 を下回り、オーケストレーションの方が高くつく
- 設計判断・プロジェクト全体の文脈が要る作業・セキュリティ絡み
- 急ぎの作業(実時間は 3.7〜4.1 倍。うち構造上避けられないのは ~1.5 倍、残りはローカル推論速度)

**非同期(submit/wait)を使う場面:** 原則、headless の自動委譲では使わない。単一タスクでも(+33%)、委譲向き A を投げて別作業 B を自分でやる本来の使い方でも(B≪A のため net 壁時計は縮まず)得はなかった。価値があるのは、人間が対話セッションでローカルの作業と無関係な同規模の別作業を並行するときだけ。

**設定の書き方で分かったこと:**
- 「常に委譲せよ」というソフトな指示は**半分無視される**(しかも損益分岐近傍ではその不服従が合理的)。指示は「いつ委譲が得か」の判断基準として書く方が実態に合う
- コスト以外の理由(プライバシー・オフライン)で委譲を必須にしたい場合は、「最初の呼び出しが返る前にファイルを編集しない」レベルの外せない表現まで強めると遵守が 3/6 → 3/3 に上がった
- ローカル実行は 1〜20 分かかる。**既定 2 分のツールタイムアウトはワーカーを殺す**ので ≥900 秒を明示する

## 落とし穴コレクション

- **ネストしたエージェント CLI をバックグラウンド実行すると静かに起動死する**(環境による)。タスク状態は "running" のままプロセスが不在、というやっかいな壊れ方をする。40 分×2 回を溶かして学んだ。フォアグラウンド+長タイムアウトが安全
- **CLI の自動更新が実測のドルを別物にする**。Claude Code 2.1.77→2.1.202 でシステムプレフィックスが約3.3倍・キャッシュ書き実効単価が1.6倍になり、同じ実験の「昨日の $0.19」と「今日の $0.53」が比較不能になった。バージョンを記録し、比較は同日・同バージョン・キャッシュ warm 統制で取り直すこと。cold/warm だけでもターン1で ~$0.14 ずれる
- **ネストした claude が「長い Bash 呼び出し直後の初回 API コール」で間欠的に ConnectionRefused 死する環境があった**(外部 curl は同時刻も常に到達可能=プロセス局所。sandbox・シム・アイドル長は無関係、根本原因は未特定でEDR容疑)。対策は「作業の生存をワーカー側の永続化に寄せる」こと——`lh batch` はタスク完了毎に逐次保存するので、呼び出し元が死んでも成果と check 結果が残り、feedback は事後に記録できた
- **プロンプトトークン数の比較は罠**: `input + cache_read + cache_creation` の合計はコストと比例しない(キャッシュ読出は激安)。比較は必ず `total_cost_usd` で
- **n=1 の実測は方向性の根拠**であって統計的有意差ではない。今回は効果が桁で出る設計(コスト差・PASS/FAIL)にして回避した
- 評価ランナーには**改ざん検出**(テストファイルの sha256 スナップショット)を入れておくこと。エージェントは悪意なくテストを「修正」しにくる

## まとめと今後

「ローカル LLM への委譲でトークン節約」という設計目標は、**正しく限定すれば実測で成立する**: 多ターンの機械作業+厳密検証+ローカルワーカー、という条件下で −22%・品質退行ゼロ。さらに第5ラウンドで、独立タスクを 1 セッションにまとめて委譲すれば 1件あたり $0.064(単発比 −52%)まで下がり、単発では損する軽タスクもバッチなら黒字化することが分かった。一方で、委譲の経済性を支配しているのは委譲先の性能ではなく**呼び出し元の固定費構造**——しかもその大半は「セッション 1 本を起動する費用」だった、というのが一連の学びだ。

第4ラウンドで予告した 2 つの追試は、どちらも予想を裏切る形で決着した。(1) 委譲が最も効くはずの重量級タスクは、**baseline 側が Python スクリプトに畳んで安く済ませてしまい、予告した「−80% 級」は幻だった**(予測 $1.1〜1.5 → baseline 実測 $0.23、委譲 $0.129 で −44%)。委譲は下限の「固定費の壁」と上限の「スクリプト化の天井」に挟まれ、現実的な節約は −30〜50% 帯が上限——狙い所は「委譲 vs スクリプト化 vs 自力」の三択で考えるべきだった。(2) submit/wait 非同期モードは、単一タスクではブロック時間を 1 秒も回収せず +33% の純損だった。

第5ラウンドはこの 2 つに決着をつけた。**バッチ委譲は当たり**——固定費の正体がセッション起動費だと判明し、束ねれば軽タスクも黒字化する。**submit/wait は外れ**——本来の使い所(委譲向き A ‖ 別作業 B)でも B ≪ A のため net 壁時計は縮まず、headless では「使わない機能」と断定できた(価値は人間の対話セッションでの並行作業のみ)。あわせて、書きっぱなしだった feedback を読む学習ループを閉じた: 委譲判断の前に `lh stats --by-kind` でその種類の合格率を見て較正する。

第6ラウンドはバッチ委譲を一級コマンド `lh batch` に昇格させ、その途中で Claude Code の自動更新に物差しごと変えられた。新会計(2.1.202)の同日 3-way では、**手組みの逐次委譲はベースラインに負ける赤字に転落し、1 コールに畳む `lh batch` だけが黒字を維持**(−6〜13%、対手組み ≈−30%)。委譲済みタスクの逐次永続化は、測定当日に多発した間欠ネットワーク断からも実験を守った。「ドルの実測は同日・同バージョン・warm 統制でしか比較できない」は、この種の測定をする全員に効く教訓だと思う。

残る宿題は、(a) check 失敗時に同一文脈へ追撃できる `--resume` の実効性を実際の差し戻し例で測ること、(b) ローカル推論の実時間短縮(委譲の最大ペナルティはコストでなく壁時計で、`num_ctx` の見直し余地がある)+ 一級 batch のローカル実行が手組み 3 連発より 2〜5 倍遅かった原因の切り分け、(c) stats 事前チェックの効果の定量測定(fail 過半の kind が実データにたまってから)、(d) 公開前に見出し数値(全セル n=1)を**同日・同バージョン・warm 統制で**数回ずつ再測して幅を併記すること。
